# OctoAcme Project Management Docs

Welcome to the OctoAcme Project Management Docs! This repository contains collaboratively-developed, versioned guidance for running projects at OctoAcme.

## OctoAcme Project Management Overview

OctoAcme follows a structured, evidence-based approach to project management built on core principles that prioritize customer value, team transparency, and continuous improvement.

### Core Principles

- **Customer-first:** We prioritize customer value and usability in all decision-making
- **Iterative delivery:** We deliver small, testable increments rather than large, monolithic releases
- **Clear ownership:** Every project has a named Project Manager (PM) and Product Manager (PdM) with defined responsibilities
- **Data-informed decisions:** We measure impact and iterate based on evidence and user feedback
- **Psychological safety:** We encourage feedback, learning, and blameless retrospectives
- **Risk management discipline:** We proactively identify, assess, and mitigate risks throughout the project lifecycle

### Key Roles

- **Project Manager (PM):** Coordinates delivery, manages schedules, risks, and communications
- **Product Manager (PdM):** Defines outcomes, prioritizes the backlog, and measures success
- **Developers:** Implement features, collaborate on design and testability
- **QA/Testing:** Validate quality and acceptance criteria
- **Stakeholders:** Provide inputs, approvals, and strategic direction

### Project Lifecycle

Every OctoAcme project follows a five-phase lifecycle:

1. **Initiation:** Define the problem statement, identify stakeholders, establish high-level timeline
2. **Planning:** Break work into shippable increments, define acceptance criteria, identify dependencies and risks
3. **Execution:** Build, test, review, and iterate with regular standups and progress tracking
4. **Release:** Deploy features to production with proper testing, rollback plans, and stakeholder communication
5. **Retrospective & Continuous Improvement:** Capture learnings and convert them into actionable improvements

### Communication Cadence

- **Daily:** Team standups (15 min) focused on progress, blockers, and dependencies
- **Weekly:** PM and PdM alignment sync; delivery team standups (twice-weekly or as agreed)
- **Monthly:** Stakeholder updates and status reports
- **Ad-hoc:** Escalations and incident communications as needed

## Contributing to These Docs

We continuously refine and improve our processes based on team experience and feedback. To propose an update:

1. Open an issue using the **"Add Content to Project Management Process Docs"** template
2. Describe the gap or improvement needed
3. Provide suggested content (optional but helpful)
4. Reference stakeholders who should review
5. Submit a pull request with the update, linking to the issue

All documentation updates should:
- Align with existing process docs and principles
- Close a documented gap or improve clarity
- Have input from relevant stakeholders when needed
